## 1. Overview

SPGP is a Laravel 13 + React application for a private ski-pass group purchase. Public users can register only with an invite code, authenticate with a session cookie, request passes for themselves/family, ask Markdown Q&A questions, and later receive bearer-value promo codes. Admins manage users, seasons, pass types, invite codes, pass requests, a promo-code repository, email sending, and email logs.

The highest-value assets are admin sessions, the user table, pass request PII (names, emails, dates of birth, renewal/pass/order IDs), promo codes, invite codes, and email logs. Promo codes should be treated like redeemable secrets; email logs intentionally store code bodies, so they are also sensitive. The real-world confidentiality goal includes not publicly disclosing discount-program details, not just preventing account takeover.

Robustness/security measures include Laravel session authentication, password hashing and `PasswordRule::defaults()`, password-reset token expiry/throttling, session regeneration on login, an `admin` Gate in `AppServiceProvider`, route groups using `auth` and `can:admin`, Eloquent parameter binding, validation rules on most HTTP inputs, hidden `password`/`remember_token` fields in `User`, ULID-like pass request IDs, soft deletes for seasons/invites/questions, FK constraints, Blade escaping, React text escaping, some JSON_HEX encoding for dashboard data, session `HttpOnly`/`SameSite=Lax` defaults, Sentry PII disabled by default, and CSP nonces configured.

Important caveats: CSP is report-only in `config/csp.php`, `/api/*` is exempted from CSRF validation in `bootstrap/app.php` even though API routes use session auth, and user Markdown is rendered without explicit sanitization.

## 2. Threat model, Trust boundaries and assumptions

**Attacker-controlled inputs:** unauthenticated web requests to login, registration, forgot/reset password; guessed/borrowed invite codes; authenticated regular-user API calls; pass request fields; Q&A Markdown; upvotes; invite redemption; password changes; request IDs and season/pass-type IDs in URLs; browser-originated CSRF attempts; and all user-generated database content later rendered to admins/users.

**Admin/operator-controlled inputs:** admin dashboard CRUD for users, seasons, pass types, invite codes, promo-code TSV imports, promo-code assignment, email filters, `spreadsheet_url`, deployment `.env`, APP_KEY, database/mail/Sentry/AWS/session/CSP settings, and production admin accounts. Admins are trusted with broad PII/code access, but admin-entered data still crosses a web-rendering boundary and should be URL/HTML safe.

**Developer-controlled inputs:** migrations, tests, seeders, local build scripts, Vite assets, CI workflows, package locks, and artisan commands. Bugs here usually affect production only through supply-chain/deployment compromise, not direct web attacker input.

**Boundaries:** guest-to-authenticated; authenticated-user-to-admin (`can:admin`); user-owned pass requests vs other users; season/invite access boundaries; browser DOM vs stored database content; Laravel app vs database; Laravel app vs Brevo/mail, Sentry, and S3; CI runner/deploy secrets vs production host. Severity should be calibrated to whether an attacker can cross one of these boundaries remotely.

Assume the app is internet reachable but used by a small invited community. Availability bugs are less severe than PII/promo-code compromise, except near purchase deadlines. Admin compromise is catastrophic because admin features intentionally allow viewing/modifying all PII and codes and impersonating users.

## 3. Attack surface, mitigations and attacker stories

**Authentication and sessions.** `AuthController` handles login, invite-code registration, password reset, password change, logout, and admin impersonation. Passwords are hashed and sessions regenerate on login/logout. Password reset uses Laravel expiry/throttle. Missing explicit login/register/forgot-password rate limiting makes credential stuffing, password spraying, invite-code guessing, and reset-email flooding important to test. Registration checks invite `max_number_of_uses`, but the check/attach is not transactional and the database has no quota constraint, so concurrent registrations may overuse a code. Admin impersonation is gated and logged in `user_logins`, but it replaces the admin session; any XSS/CSRF that reaches it can pivot into user accounts.

**Authorization and IDOR.** Admin APIs are grouped under `can:admin`; user pass-request update/delete checks owner or admin. However, several object-level checks are security-sensitive: `PassRequestController::store` validates `season_id` and `season_pass_type_id` independently but does not require the pass type to belong to the season, does not reject archived/closed/future-inaccessible seasons, and does not enforce deadlines server-side. `showRequestForm` and `getPassTypes` have little season access logic. `QuestionController` API methods (`getQuestions`, `store`, upvote/unvote) do not repeat the season access check used by the web view, so any authenticated user may read/post against arbitrary season IDs. These are high-value review targets because they affect confidential season details and purchase integrity.

**CSRF.** Frontend requests include `X-CSRF-TOKEN`, but `bootstrap/app.php` exempts `api/*` from CSRF validation while `routes/api.php` uses `web` sessions. Browser `SameSite=Lax` cookies reduce many cross-site POST attacks, and CORS prevents reading responses, but SameSite is not a server-side authorization control and may be weakened by configuration, legacy clients, or same-site subdomain issues. A reliable admin CSRF could promote users, change accounts, assign/clear codes, send emails, delete requests, or impersonate.

**Stored XSS/frontend security.** Most plain text rendering is protected by Blade/React escaping. Dashboard JSON uses `JSON_HEX_TAG|JSON_HEX_AMP`. The Q&A feature renders untrusted Markdown in `resources/js/components/Questions.tsx` using `MDEditor.Markdown` without `rehype-sanitize`; the library documentation warns this is unsafe for untrusted authors. Because any authenticated user can post questions and admins view unanswered questions, stored XSS can become critical. CSP nonces exist, but the policy is currently report-only, so it will not block payloads.

**Data validation, injection, and integrity.** Laravel validation and Eloquent greatly reduce SQL injection risk, including `LIKE` filters in `EmailLogController`. Inputs generally have length/type constraints, but Q&A content and promo-code TSV imports are unbounded enough for storage/browser DoS. Duplicate pass requests are checked in application code but not protected by a unique index, so races or later edits can create duplicates. Promo-code assignment should be reviewed for race conditions because codes are unique secrets but assignment is not wrapped in transactions/locks.

**Admin, emails, and privacy.** Admin endpoints intentionally expose all users, pass requests, promo codes, and email logs. `SeasonController::sendEmails` emails the passholder and the admin performing the action; the admin mailbox therefore receives many codes. Email logs store promo codes and mail errors. Treat log access/backups as sensitive. `spreadsheet_url` is only followed by the admin browser, so SSRF is not present, but server-side validation should still restrict to `https://` to avoid phishing or dangerous schemes.

**Spreadsheet/clipboard workflows.** Admin UI copies TSV containing attacker-controlled names/emails plus DOBs and promo codes. If pasted into Excel/Sheets, values beginning with `=`, `+`, `-`, or `@` can become formulas and exfiltrate data. This is unusually relevant because the README documents spreadsheet-based operations.

**External services and deployment.** There is no production route using `FileStorageService`, so file upload/path traversal issues are latent unless future routes expose attacker-controlled S3 paths. SSRF is mostly out of scope because user URLs are not fetched server-side. CI/deploy is not runtime code, but `.github/workflows/deploy.yml` uses SSH password auth and `StrictHostKeyChecking=no`; a CI/network compromise could redirect deployment or expose secrets.

## 4. Criticality calibration (critical, high, medium, low)

**Critical:** remote code execution; SQL injection dumping/modifying users, pass requests, promo codes, or sessions; authentication bypass to admin; stored XSS reachable by admins that can call admin APIs or exfiltrate promo codes/PII; leakage of APP_KEY/session secrets enabling session forgery; reliable admin CSRF leading to privilege escalation or mass code/email destruction.

**High:** IDOR allowing users to read/modify other users pass requests or promo codes; bypassing `can:admin`; creating/altering pass requests for closed seasons or mismatched pass types at scale; brute-forceable login/invite flows without throttling; promo-code race causing duplicate assignment or theft; spreadsheet formula injection that can exfiltrate admin-opened TSV data; persistent XSS affecting regular users only.

**Medium:** unauthorized reading/posting of Q&A for inactive or unrelated seasons; invite-code overuse race; unbounded Markdown/TSV causing DB/browser/mail DoS; email/user enumeration; weak server-side URL validation for admin-only links; duplicate pass-request integrity bugs without code theft; Sentry/log misconfiguration leaking limited PII.

**Low:** UI-only inconsistencies, harmless validation gaps on admin-only fields, local artisan command footguns, test/fixture issues, CSP remaining report-only without an exploitable injection, minor information messages, or deployment/documentation hardening that requires operator access. File-upload, deserialization, SSRF, and command-injection findings should not be rated high unless a production route gives an attacker the needed control.
